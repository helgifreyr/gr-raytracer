The Engineering

A general-relativistic ray tracer that runs in real time, in a browser tab, with no plugins — one independent light ray integrated per pixel, in parallel, on your GPU.

It all runs on the GPU

The heavy lifting is a WGSL fragment shader (src/shaders/scene.wgsl) that the GPU executes once per pixel, thousands of cores at a time. Each invocation builds a photon, integrates its geodesic through curved spacetime over hundreds of steps, figures out what it hit, and returns a colour. A 1080p frame is ~2 million independent geodesic integrations running simultaneously. The JavaScript host (src/main.js, on the CPU) does no per-ray physics — it sets up WebGPU, reads the UI, packs the camera and parameters into a uniform buffer, and issues draw calls.

The trick: autodiff instead of Christoffel symbols

The geodesic equation needs the derivatives of the metric. Traditionally you'd hand-derive the Christoffel symbols for each spacetime — tedious and error-prone. Instead this uses forward-mode automatic differentiation with dual numbers. A dual number carries a value and its gradient, $v + \mathbf{d}\,\varepsilon$ with $\varepsilon^2 = 0$, and any function propagates it exactly: $f(v + \mathbf{d}\,\varepsilon) = f(v) + f'(v)\,\mathbf{d}\,\varepsilon$. Seeding each coordinate with a basis-vector gradient and evaluating the metric once yields both $g^{\mu\nu}$ and every partial $\partial_\alpha g^{\mu\nu}$ the equations of motion need (struct Dual { v: f32, d: vec4<f32> } — the value plus a 4-gradient). This is the idea from FANTASY (Christian & Chan 2020), recast as a real-time GPU shader.

Consequence: the metric is the only spacetime-specific code. Everything downstream — the right-hand side, the RK4 integrator, the camera — is generic. Adding a new black hole is one new branch returning gᵃᵇ as dual numbers; nothing else changes.

Integrator and capture

The 8-component state (qᵃ, pₐ) is advanced with classic RK4, with the step size shrinking in the strong field so the tight windings near the photon orbits are resolved. A ray ends in one of three ways: captured (black), it crosses the disk slab (emission), or it escapes and samples the lensed background.

Capture is metric-derived, never a hard-coded radius: it's each spacetime's actual event horizon. For static slicings the redshift −g⁰⁰ = 1/lapse² blows up at the horizon; for the horizon-penetrating Kerr–Schild coordinates the metric is regular there, so we use the Kerr radial coordinate r < r₊ (exact at every latitude — using the Euclidean radius instead leaks rays through the shadow off-equator, a bug we hit and fixed).

The accretion disk is volumetric

The disk is a Gaussian slab integrated as emission + absorption along each ray (sub-sampled per segment, independent of the geodesic step), so a thick, edge-on disk stays smooth and softly self-occluding rather than being an infinitely thin plane. Each segment advances the radiative-transfer equation $\frac{dI}{ds} = \kappa\,(S - I)$ — emission toward a source function $S$, attenuation by opacity $\kappa$ — so an optically thick disk emerges at its source colour. The relativistic Doppler factor and the gravitational lapse are combined into one redshift factor that drives the beaming (∝ δ³) and the observed temperature. The source colour is a Shakura–Sunyaev profile $T \propto r^{-3/4}$ indexing a stylized hot→cool palette, and the gas is given filamentary structure by multifractal value-noise sampled in log-spiral coordinates (technique adapted from cuneus, MIT).

HDR and bloom

The scene renders to a floating-point HDR buffer; a second shader (src/shaders/post.wgsl) extracts the bright parts and blurs them into a three-octave pyramid (½, ¼, ⅛ resolution) with separable Gaussians, then composites the octaves back for a soft, band-free glow. The final step is a hue-preserving ACES filmic tonemap (the ACES curve applied to the luminance peak, chroma ratio preserved) and gamma — so the beamed side blooms without the colour clipping plain per-channel Reinhard would cause.

The f32 ceiling

WebGPU shaders are single-precision only (no f64). The physics is correct, but near the horizon and photon sphere — where geodesics are stiff — f32 shows its limits: near-extremal Kerr (a → 1) leaves faint shimmer at the very edge of the shadow. Horizon-penetrating coordinates and strong-field step refinement keep it well-behaved everywhere else.

How it's validated (without a GPU)

You can't run WebGPU headless in CI, so correctness is established on the CPU. There's a double-precision JavaScript twin of the geodesic engine (test/engine.mjs) checked against analytic GR — light deflection (including the second-order term), Hamiltonian conservation, frame-dragging asymmetry, horizon capture, the wormhole thread/reflect dichotomy. Then the shipped WGSL is extracted and run through a CPU interpreter and compared against that twin (test/wgsl_exec.mjs), including the Kerr off-diagonal terms. npm test runs 21 physics checks plus 17 shader-vs-engine checks. (What's not auto-tested: real-GPU compilation, the host's uniform packing, and the visual look — those are confirmed by opening the page.)

Architecture

index.html              page shell (canvas + control panel + this menu)
src/main.js             WebGPU host: device, pipelines, textures, render loop, UI
src/shaders/scene.wgsl  the geodesic engine + shading — runs per pixel on the GPU
src/shaders/post.wgsl   HDR bloom passes
test/                   Node-only CPU twin + validation (never runs in the browser)

It's a static site — no build step. The shaders load as plain text files at startup, so it needs serving over http (GitHub Pages, or a local npm run serve). Rendering is on-demand: the GPU only draws when the camera or a control changes, then idles, so a still frame costs almost nothing.
